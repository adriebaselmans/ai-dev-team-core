== Purpose

This document explains how the AI Development Team System works as an operating model.

It describes:

- the agents and their responsibilities
- how the coordinator drives work across phases
- how repository exploration fits in as a conditional shared support capability
- how the runtime contract supports native subagent execution
- how artifacts move from requirements to delivery
- how memory, review, and testing keep the system durable and auditable

== System Summary

The Development Team System is a coordinator-led, phase-based delivery model for AI-assisted software work.

At a high level:

- the *Coordinator* is the only user-facing agent
- specialist agents own bounded work products
- repository exploration is a shared support capability when repository grounding is required
- each major delivery phase has an explicit artifact
- the runtime contract describes the team, workflow, validation, and task payload shape
- project memory is updated after each completed phase

== Shared Support Capability

Repository grounding is handled as a shared support capability, not as a standalone role.

The coordinator can invoke it when the work must be grounded in another codebase.

== Core Components

=== Coordinator Layer

The coordinator owns orchestration, phase transitions, memory updates, and user-facing communication.

=== Specialist Layer

Each specialist has a narrow mission and a bounded write scope.

[cols="1,2,2",options="header"]
|===
| Agent | Primary Responsibility | Primary Outputs

| Coordinator
| Orchestration, user communication, phase control, memory updates
| `framework/runtime/state.json`, `framework/memory/*`

| Repository exploration support
| Analyze a target repository and capture reusable repository knowledge
| `framework/memory/repository-knowledge/*`

| Requirements Engineer
| Convert a user need into an implementation-ready baseline
| `doc_templates/requirements/current.yaml`

| Architect
| Produce the simplest safe design that satisfies the requirements
| `doc_templates/design/current.yaml`

| Developer
| Implement the design and add unit-level verification
| `src/`

| Reviewer
| Perform technical review for correctness and maintainability risks
| `doc_templates/review/current.yaml`

| Tester
| Validate user-visible behavior and prepare delivery evidence
| `doc_templates/dod/current.yaml`
|===

=== Runtime Contract Layer

The `framework/runtime/` directory is the machine-readable contract for orchestration.

The repository exploration capability is a shared support path the coordinator can invoke when work must be grounded in another codebase.

== Skills and Execution Guidance

- the coordinator uses `coordinator-flow`, `memory-update`, and `compaction`
- the repository exploration support capability uses `repository-exploration` and `repository-knowledge-compaction`
- the requirements engineer uses `requirements-clarification`
- the developer uses `implementation-clean-code` and `unit-testing`
- the tester uses `acceptance-testing`

== Approval and Autonomy Model

The workflow uses a constrained autonomy model: the coordinator may invoke shared support when grounding is required, and after requirements are clear the system should continue through architecture, development, review, and testing without extra approval checkpoints.

== Conclusion

The Development Team System turns agent collaboration into a governed workflow with clear ownership, durable artifacts, explicit validation, and controlled autonomy.
